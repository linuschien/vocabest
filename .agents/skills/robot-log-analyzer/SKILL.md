# Skill: Robot Framework Test Result & Defect Analyzer

## ℹ️ Objective
A utility that parses the XML output generated by Robot Framework, extracts failure traces, resolves test cases back to original Gherkin specifications and requirements, and compiles a standardized markdown E2E audit report.

## 🛠️ Analyzer CLI Arguments
The underlying script `analyze_failures.py` accepts the following inputs:
- `--xml`, `-x`: Path to the input Robot Framework `output.xml` file. (Default: `engineers/04-tests/reports/output.xml`)
- `--output`, `-o`: Path to save the final Markdown report. (Supports `{timestamp}` which is resolved to `YYYYMMDD_HHMMSS`. Default: `engineers/04-tests/reports/e2e-report-{timestamp}.md`)
- `--features-dir`, `-f`: Path to user behavior spec `.feature` files to parse. (Default: `docs/02-design-specs/behavior-specs/user/`)
- `--requirements-dir`, `-r`: Path to requirements `.md` files. (Default: `docs/01-requirements/user-stories/`)

## ⚙️ Parsing and Mapping Logic

### 1. Metrics Calculation
- **Total Test Cases**: Total count of `<test>` tags in the XML.
- **Passed Cases**: Count of `<test>` tags where `<status>` is `PASS`.
- **Failed Cases**: Count of `<test>` tags where `<status>` is `FAIL`.
- **Pass Rate**: `(Passed / Total) * 100` formatted to 2 decimal places.
- **Duration**: Extracted from the root `<suite>` status elapsed time or computed from start/end times.

### 2. Metadata Extraction from Robot Files
- For each test suite, the parser reads its `.robot` source file (path found in the `source` attribute of the suite).
- It extracts:
  - `Source Feature`: Mapped from `Source Feature : <path>` in the suite documentation.
  - `Upstream Req`: Mapped from `Upstream Req : <filename>` in the suite documentation.
- Using these, the parser constructs the **Traceability Matrix** with relative markdown file links.

### 3. Defect Analysis
For every failed test case, the script:
1. Traces the full suite hierarchy path.
2. Identifies the Gherkin step/keyword that failed by locating the deepest nested `<kw>` with a status of `FAIL`.
3. Extracts the exact failure error message from the `<msg level="FAIL">` element inside the failed keyword.
4. Categorizes the defect under its parent suite.
5. Formats the output with detailed expected vs. actual analysis sections.

## ⚠️ Hard Constraints
- **Do not modify source code**: The analyzer script must only read outputs and write reports; it must never modify backend/frontend source files.
- **Output Path Validation**: The generated Markdown report must be saved strictly under `engineers/04-tests/reports/`.
- **File Link Accuracy**: All file links in the Traceability Matrix must be absolute links using the `file:///` URI scheme.
